# CoreKnowledge 管理端与用户端重组设计

## 目标

把当前已跑通的检索实验页面重组为可长期使用的正式产品：一个管理员端负责知识库路径、模型、用户和接入配置；一个用户端供匿名、外部和内部用户通过 Web 或飞书完成原文查询与知识问答。

现有最小闭环不废弃：文档解析、增量扫描、RAGLite 混合检索与重排、PostgreSQL + pgvector、DeepSeek/Ollama 适配和隐私策略继续作为底层能力。前端在现有 React 工程内演进，不另建预览项目。

## 身份与权限

| 身份 | 形成方式 | Web | 飞书 | 知识范围 |
| --- | --- | --- | --- | --- |
| 匿名用户 | 未绑定账号 | 可用 | 不适用 | 公开知识 |
| 外部用户 | 飞书绑定成功或识别失败 | 可用 | 可用 | 公开知识 |
| 内部用户 | 管理员明确授权 | 可用 | 可用 | 公开与敏感知识 |

绑定不等于内部授权。匿名历史只保存在浏览器；绑定时必须经用户确认后才能合并到账号。身份降级后，敏感答案、原文、引用和文档预览立即不可见。

## 两种核心能力

原文查询不调用大模型，返回命中的原始片段、文件名和定位信息。知识问答必须先检索，再由获准使用的模型根据证据总结或汇总，并显示引用；证据不足时明确拒答。

当证据包含敏感内容、当前模型为云端且“允许敏感内容发送云端”关闭时，不调用云端模型，只返回本地原文片段。切换到已配置的本地模型后，才允许对敏感证据生成总结。

## 信息架构

用户端固定为 `/app/home`、`/app/search`、`/app/chat`、`/app/shortcuts`、`/app/history`、`/app/documents`、`/app/bind-feishu`、`/app/profile`。

管理端固定为 `/admin/login`、`/admin/dashboard`、`/admin/knowledge`、`/admin/knowledge/scan-detail`、`/admin/models`、`/admin/users`、`/admin/records`、`/admin/analytics`、`/admin/shortcuts`、`/admin/feishu`、`/admin/audit`、`/admin/retrieval-lab`。

管理员可以查看全局查询与问答次数、记录和用户活跃度；普通用户只能查看自己的次数与历史。快捷入口由个人高频行为与管理员配置共同组成，并严格区分“查询模板”和“问答模板”。

## 飞书交互

飞书只输出纯文本，不使用菜单或卡片。支持 `查询：...`、`问答：...`、`历史`、`帮助`，无前缀文本默认进入知识问答。回答文本包含简短结论或原文、来源与必要的权限/隐私提示。

## UI 交付边界

正式视觉源为 `advanced_llm_apps/private_knowledge_customer_service/docs/ui/stitch/stitch_coreknowledge/`。画板用于还原布局和视觉，不覆盖业务规则；目录内 `README.md` 的修正规则优先于画板中的旧英文、固定身份、上传按钮或错误模型描述。

## 实施顺序

1. 建立正式路由、用户端和管理端外壳及动态身份上下文。
2. 接通原文查询、知识问答和引用，保留现有检索实验室作为管理员调试页。
3. 接通知识库扫描、模型设置与隐私策略。
4. 实现绑定、用户授权、历史、次数和快捷入口。
5. 接入飞书纯文本协议，再完成全局审计与数据分析。

每一步都必须使用真实 API 和现有底层能力验收，不制作一次性预览页面。
