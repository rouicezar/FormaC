## 全视图与重点区域结论

- 信息架构：用户端采用窄图标导航、身份顶栏、双核心入口；管理端采用完整侧栏、指标卡、服务状态和快捷管理，保持冻结画板的主层级。
- 字体：使用苹方、冬青黑体、微软雅黑系统中文字体栈；标题、正文、标签的字号与字重层级清晰，无英文模板字体依赖。
- 布局与间距：桌面端卡片网格、边距、圆角和细边框与冻结稿一致；移动端无水平溢出，核心入口单列排列，底部导航保持可操作。
- 色彩：使用米白背景、白色表面、低饱和墨绿和深灰绿文字；无纯黑大面积背景、荧光色、强渐变或发光效果。
- 图片与图标：冻结页面没有业务图片资产；正式实现沿用工程已有 Lucide 图标库，图标尺寸和线条保持一致，没有使用占位图或 CSS 绘图冒充资产。
- 文案：所有产品文案为简体中文；明确区分“不调用大模型的原文查询”与“基于知识库证据总结的知识问答”。
- 状态与交互：导航路由、首页核心入口、管理首页刷新按钮、加载状态、后端状态、待接入指标和占位路由均可工作。
- 可访问性：导航带语义标签；按钮、链接和焦点样式可由键盘识别；颜色不作为服务状态的唯一信息。

## 有意偏离冻结稿的内容

- 冻结稿中的固定“系统管理员/内部用户”已改为动态身份，默认匿名访客，符合权限铁律。
- 冻结稿中的虚构查询次数、问答次数、用户数、审计事件和高频词被移除；尚无真实 API 的指标显示“待接入”。
- 用户首页的最近记录与快捷入口使用真实空状态，未照搬演示数据。
- 管理首页系统连接状态直接读取 FastAPI `/health`，而不是使用静态成功状态。
- Stitch 中“新建数据源”入口未实现，因为系统只读取管理员授权的本地目录。

上述偏离均属于已确认业务规则对视觉稿演示内容的修正，不是设计遗漏。

## 浏览器验证

- `/app/home`：匿名身份、公开知识范围、双入口和导航均可见。
- `/admin/dashboard`：真实健康检查返回 4/6 个依赖运行正常；自动扫描显示尚未启动，飞书显示尚未配置。
- 点击“刷新状态”后恢复“检查完成”。
- 浏览器控制台未发现错误或警告。
- 390 × 844 下页面宽度与视口均为 390px，没有横向滚动。

## 后续轻量优化

- Stitch 使用深墨绿突出知识问答卡；当前实现采用两个同权白色入口，避免用户误解查询与问答存在主次。这属于可在用户体验反馈后调整的 P3 视觉选择。
- 等真实统计接口完成后，用真实数据替换管理首页的“待接入”，并补充趋势与时间范围。

## 对照历史

首轮对照未发现 P0、P1 或 P2 问题，因此没有阻断性修复循环。动态身份、真实空状态和待接入指标均在实现前由产品铁律明确要求，已作为有意差异验收。

## 第二阶段：原文查询与知识问答

- 原文查询设计源：`docs/ui/stitch/stitch_coreknowledge/app/search-results/screen.png`
- 原文查询实现截图：`docs/ui/implementation-app-search.png`

- 原文查询组合对照：`docs/ui/qa-app-search-comparison.png`
- 知识问答设计源：`docs/ui/stitch/stitch_coreknowledge/app/chat-answer/screen.png`
- 知识问答实现截图：`docs/ui/implementation-app-chat.png`
- 知识问答移动截图：`docs/ui/implementation-app-chat-mobile.png`
- 知识问答组合对照：`docs/ui/qa-app-chat-comparison.png`
- 桌面实现视口：1513 × 862；设计源经过等宽归一化后并排对照。
- 移动实现视口：390 × 844。

### 验收结论

- 原文查询保留冻结稿的隐私承诺、搜索框、结果列表与原文预览双栏结构，明确显示“未调用模型”。
- 知识问答保留问题、回答、引用侧栏和底部输入区，真实展示 Ollama 生成结果与三条知识库引用。
- 查询、问答加载状态、查询默认态、查询无结果态、问答错误态、证据不足态和敏感云端降级态均有独立组件逻辑。
- 390 × 844 下两页 `scrollWidth` 与 `clientWidth` 均为 390px；导航固定在底部，页面纵向滚动，不存在水平裁切。
- 真实交互验证了查询输入、提交、结果选择、原文预览、问答输入、模型选择、回答生成和引用展示。
- 浏览器控制台没有错误或警告。

### 有意差异

- Stitch 查询稿包含全局筛选与虚构文件；本阶段只实现真实后端支持的查询词和权限范围，不制作无 API 的静态筛选器。
- Stitch 问答稿包含英文工作区、收藏和附件功能，均不属于当前冻结范围，正式实现使用统一中文用户导航。
- 模型选择只展示当前已接入的 Ollama 与 DeepSeek；敏感云端选择框仅对内部身份和云端模型出现，且默认关闭。

第二阶段对照未发现 P0、P1 或 P2 问题；上述差异均来自已冻结业务规则和真实 API 边界。残余 P3 为桌面宽屏上的内容留白，后续接入多轮历史后自然填充，不阻断当前验收。

## 第三阶段：知识库与模型配置

### 对照依据

- 知识库设计源：`docs/ui/stitch/stitch_coreknowledge/admin/knowledge/screen.png`
- 模型配置设计源：`docs/ui/stitch/stitch_coreknowledge/admin/models/screen.png`
- 敏感策略确认设计源：`docs/ui/stitch/stitch_coreknowledge/admin/models-sensitive-confirm/screen.png`
- 桌面真实路由：`/admin/knowledge`、`/admin/models`
- 移动验收视口：390 × 844。

### 验收结论

- 两页沿用正式管理侧栏、米白画布、白色配置卡、低饱和墨绿操作色和细边框，未引入平行页面或模板英文侧栏。
- 知识页以唯一根目录呈现公开/敏感固定分区，手动扫描直接调用真实 API，并展示真实统计和失败态。
- 模型页清楚分隔 DeepSeek 云端与 Ollama 本地插件；API Key 输入不回显已有秘密；原文查询不经过模型的边界在页首明确说明。
- 敏感云端开关默认关闭，保存开启状态会出现独立二次确认对话框；取消后开关恢复关闭，后端仍会再次校验确认字段。
- 真实浏览器已验证配置读取、原值保存、手动扫描、确认弹窗、取消、安全保存和控制台零错误。
- 390 × 844 下两个页面均无水平溢出；配置卡改为单列，扫描指标与操作按钮保持可用。

### 有意差异

- Stitch 知识页中的上传、新建数据源与任意双路径编辑违反“只读取授权根目录”规则，正式实现改为一个根目录和两个只读派生路径。
- Stitch 模型稿中的虚构延迟、连接健康度和未实现的数据脱敏开关没有进入产品；页面只展示已有真实模型插件和隐私策略。
- 正式实现没有在用户问答页暴露敏感云端开关，避免普通请求控制管理员级策略。

第三阶段未发现 P0、P1 或 P2 视觉或交互问题。

## 第四阶段：用户管理与身份审计

### 对照依据

- 用户管理设计源：`docs/ui/stitch/stitch_coreknowledge/admin/users/screen.png`
- 正式实现路由：`/admin/users`
- 移动验收视口：390 × 844。

### 验收结论

- 保留冻结稿的身份统计、用户表格、权限操作与最近审计结构，并统一使用正式管理侧栏、米白画布和低饱和墨绿视觉体系。
- 页面数据全部来自身份与审计 API；飞书身份绑定后明确显示“外部用户”，提权和降级均有独立确认对话框与成功回执。
- 真实浏览器完成外部绑定、内部提权、外部降级和审计顺序验证；控制台无错误。
- 390 × 844 下页面本身无横向溢出；桌面宽表在移动端保留表格语义并限制在卡片内部滚动。

### 有意差异

- 删除 Stitch 中 2,840 名用户、今日活跃、绑定率、待审核、查询次数和最后访问等虚构数据；当前数据库只展示真实已绑定身份。
- 匿名用户不进入服务端身份表；匿名历史仍属于浏览器本地数据，避免把虚构 IP 记录成用户。
- 首版只支持逐个提权和降级，不实现没有批量审计契约的批量操作。

第四阶段在真实联调中发现并修复一个 P1：用户列表与审计并发读取时共享 Session 会冲突。修复为每请求独立 Session 后，页面和真实 PostgreSQL 验收通过；没有剩余 P0、P1 或 P2 问题。
