# CoreKnowledge 项目铁律

本文件适用于 `private_knowledge_customer_service/` 下的全部需求、设计、实现、测试与交付。任何代理进入本项目后，必须先阅读本文件、`docs/requirements.md`、`docs/design.md`、`docs/project-status.md` 以及 Stitch 冻结清单。

## 工程与复用

1. 禁止为了让用户快速预览而制作一次性页面、平行前端、静态假界面或不可延续的演示形态。
2. 正式功能必须在现有 React + FastAPI 工程中持续演进，并连接真实 API；暂未实现的数据必须明确显示“待接入”，不得伪造。
3. `awesome-llm-apps` 中已经存在且满足要求的能力必须直接复用或薄适配，禁止重复开发。复用边界以 `docs/reuse-audit.md` 为准。
4. 必须遵循：需求 → 设计 → 实现 → 测试 → 提交 → 推送。测试失败不得宣称阶段完成。
5. 坚持最小必要修改，不重写已经跑通的扫描、解析、RAGLite、pgvector、DeepSeek/Ollama 和隐私策略。

## 产品规则

1. 产品由用户端 `/app/*` 与管理端 `/admin/*` 两部分组成，正式路由以 `docs/design.md` 为准。
2. 原文查询只查询原始资料，不调用大模型；知识问答必须先检索知识库，再基于证据总结或汇总，并展示引用。两者必须分开记录、统计和展示。
3. 匿名用户可查询和问答公开知识；飞书绑定成功后默认是外部用户；只有管理员明确授权后才能成为内部用户。身份识别失败一律按外部用户处理。
4. 外部与匿名用户只能访问公开知识；内部用户可访问公开与敏感知识。身份降级后，敏感答案、原文、引用和预览立即隐藏。
5. 系统只读取管理员授权的本地 `public/` 与 `sensitive/` 文件夹，不提供上传、删除、移动或修改源文件的功能。
6. 敏感内容发往云端的开关默认关闭。关闭时不得把任何敏感片段发送给云端模型，只能返回原文检索结果；配置的本地模型可以总结敏感内容。
7. 模型提供商必须可插拔，支持云端和本地模型，不得将 DeepSeek、Ollama 或某台硬件写死为唯一选择。
8. 飞书只使用纯文本交互，不设计菜单或消息卡片。协议为 `查询：问题`、`问答：问题`、`历史`、`帮助`，无前缀文本默认按问答处理。
9. 用户和管理员均可查看自己的历史及查询/问答次数；管理员还可查看全局记录和统计。快捷入口结合管理员配置与用户频次生成。

## UI 与隐私

1. 正式视觉依据是 `docs/ui/stitch/stitch_coreknowledge/`；必须先读其 `README.md` 修正清单，禁止照搬 Stitch 中的英文侧栏、固定身份、上传按钮和错误模型描述。
2. 所有用户可见 Web 文案、状态、错误、表单和无障碍标签必须使用自然的简体中文；模型名、协议名、文件名和原文可以保留原语言。
3. 使用温和米白、低饱和墨绿和深灰绿视觉体系；禁止大面积纯黑、荧光色、强渐变、发光效果和模板化 AI 演示风格。
4. 服务端权限过滤才是安全边界，前端隐藏不能代替权限检查。
5. API Key、飞书凭证、敏感原文不得写入源码、Git、浏览器日志或普通运行日志。

## 阶段验收

- 每个正式页面必须验证真实路由、导航、加载、空状态、错误状态、键盘焦点和响应式布局。
- UI 阶段必须完成生产构建与真实浏览器检查，并与冻结截图进行视觉对照。
- 每次有意义的实施后更新 `docs/project-status.md`，记录真实完成项、验证证据、未完成项和下一步。
- 只有测试通过后才能提交，并推送到 `origin`（FormaC）；模板上游保留为 `upstream`。
